# DawnPilot Agent Instructions

## Scope

These instructions apply to the entire repository. If a subdirectory gains a
more specific `AGENTS.md`, follow both files and let the more specific rules win
for that subtree.

## Product contract

DawnPilot (晨航) is an iOS 26 personal wake-up alarm for weather-sensitive
commutes. Preserve these product decisions unless the user explicitly changes
them:

- The app creates app-owned system alarms with AlarmKit. It does not add alarms
  to Apple's Clock app and does not create standalone Apple Watch alarms.
- The VPS fetches and normalizes weather. The iPhone owns user settings,
  evaluates the forecast, and chooses the alarm time locally.
- A nightly Shortcuts automation is the primary refresh trigger. BackgroundTasks
  is best effort only and must never be presented as guaranteed execution.
- The app maintains a 14-day horizon of one-off fallback alarms so a failed
  weather request or missed automation does not leave the user without an alarm.
- Location is a fixed, user-editable latitude/longitude. Do not add live location,
  APNs, holiday calendars, or address search unless explicitly requested.
- Default rules are rainy `07:50`, unknown/failure `08:00`, clear `08:05`,
  forecast window `07:00..<09:00`, precipitation threshold `40%`, Monday-Friday.
- The default server base URL may be public, but the Bearer token must remain
  empty in source and be entered by the user at runtime.

## Repository map

- `project.yml`: source of truth for targets, build settings, bundle identifiers,
  deployment target, and generated Info.plist properties.
- `DawnPilot/`: SwiftUI app, models, App Intent, weather evaluation, persistence,
  background refresh, and AlarmKit coordination.
- `DawnPilotTests/`: deterministic protocol and precipitation-rule tests.
- `server/`: dependency-free Python weather proxy, tests, systemd unit, environment
  template, and Caddy example.
- `scripts/package-altstore.sh`: canonical AltStore IPA packaging workflow.
- `DawnPilot.xcodeproj/`: generated by XcodeGen and committed for convenience.
- `build/` and `DerivedData*/`: ignored artifacts; never commit them.

## Sources of truth and generated files

- Edit `project.yml`, never `DawnPilot.xcodeproj/project.pbxproj` by hand.
- Change Info.plist properties in the target's `info.properties` block in
  `project.yml`, then run `xcodegen generate`. Do not rely on direct edits to the
  generated `DawnPilot/Info.plist`.
- Commit the regenerated Xcode project when `project.yml` changes.
- Keep these identifiers aligned everywhere:
  - App bundle ID: `com.yessicmd.dawnpilot`
  - Background refresh ID: `com.yessicmd.dawnpilot.refresh`
- Do not change the deployment target, version, build number, signing team, or
  bundle ID as an incidental cleanup.

## Architecture and correctness invariants

- `AppModel` is `@MainActor` UI state. `AlarmCoordinator` is the actor that owns
  alarm mutations and managed-alarm records. Do not bypass it from views.
- `SettingsStore` persists settings, status, and managed records in UserDefaults.
  Treat persisted Codable changes as migrations: old payloads must continue to
  decode or fall back deliberately without silently losing alarms.
- Use `AppSettings.calendar` for date, weekday, and alarm calculations. Do not use
  the process timezone for user-visible scheduling logic.
- The forecast window is half-open: start is included and end is excluded.
- A forecast is rainy when any matching hour reaches the configured probability,
  contains at least `0.1` measurable precipitation, or has a precipitation WMO
  code. A forecast older than six hours or missing target hours is a failure.
- There must be at most one managed record per local date key. Replacement must
  schedule the new alarm before canceling the old alarm and must not leave both
  records persisted after an error.
- Weather refresh failure retains the last valid alarm for tomorrow when one
  exists; otherwise it retains the fallback alarm. Never delete tomorrow's safe
  alarm merely because networking or parsing failed.
- Changing alarm times or enabled weekdays must rebuild future fallbacks.
- Keep weather and rule code testable without AlarmKit or UI dependencies.
- User-facing text is Simplified Chinese unless localization work is explicitly
  in scope.

## Swift implementation rules

- Keep UI work on the main actor and network/alarm work in async service or actor
  boundaries. Do not block the main thread.
- Preserve `Sendable` conformance for values crossing concurrency boundaries.
- Surface actionable errors to the UI. Swallow errors only for documented
  best-effort behavior such as background scheduling or rollback cleanup.
- Avoid adding third-party dependencies when Foundation and Apple frameworks are
  sufficient. Any new dependency needs a concrete benefit and user approval.
- When changing the normalized weather schema, update the Swift models, Python
  producer, protocol fixtures, and tests in the same change.
- Add or update focused tests for rule thresholds, timezone boundaries, stale
  data, missing hours, weekday selection, and fallback behavior when relevant.

## Server contract and security

- Keep the server compatible with Python 3.11 on Debian 12 and economical enough
  for 1 CPU / 1 GB RAM. It currently uses only the Python standard library.
- `GET /healthz` is public. `GET /v1/forecast` requires
  `Authorization: Bearer <token>` and must continue to use constant-time token
  comparison.
- Validate latitude, longitude, and IANA timezone before fetching upstream data.
- Preserve schema version `1` compatibility unless the app and server are migrated
  together.
- Keep the last known good forecast and serve it with `stale: true` when upstream
  refresh fails. Cache writes must remain atomic.
- Bind the Python service to `127.0.0.1:8787`; expose it only through HTTPS via a
  reverse proxy. The current public route uses the `/dawnpilot` prefix.
- Never commit, embed, echo, or log the real `DAWNPILOT_TOKEN`. Real configuration
  belongs in `/etc/dawnpilot.env` with mode `0600`; `.env.example` contains only
  placeholders.
- The cache can contain user coordinates. Preserve systemd `UMask=0077` and verify
  cache files are not world-readable.
- Do not modify a live VPS or Caddy configuration unless deployment is explicitly
  requested. Preserve unrelated routes, validate staged configuration before
  reload, and keep a rollback copy.

## Build and test commands

Regenerate the project after project configuration changes:

```bash
xcodegen generate
```

Build for the simulator without signing:

```bash
xcodebuild \
  -project DawnPilot.xcodeproj \
  -scheme DawnPilot \
  -sdk iphonesimulator \
  -destination 'generic/platform=iOS Simulator' \
  -derivedDataPath /tmp/DawnPilot-Simulator-DerivedData \
  build CODE_SIGNING_ALLOWED=NO
```

Run Swift tests using an installed iOS 26 simulator (the device name may be
changed to another available simulator):

```bash
xcodebuild test \
  -project DawnPilot.xcodeproj \
  -scheme DawnPilot \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -derivedDataPath /tmp/DawnPilot-Tests-DerivedData \
  CODE_SIGNING_ALLOWED=NO
```

Build the unsigned Release app for a physical-device architecture:

```bash
xcodebuild \
  -project DawnPilot.xcodeproj \
  -scheme DawnPilot \
  -configuration Release \
  -sdk iphoneos \
  -destination 'generic/platform=iOS' \
  -derivedDataPath /tmp/DawnPilot-Release-DerivedData \
  build CODE_SIGNING_ALLOWED=NO
```

Run server tests:

```bash
python3 -m unittest discover -s server/tests -v
```

Verification should be proportional to the change:

- Swift behavior: relevant Swift tests plus a simulator or device build.
- Alarm scheduling: rule tests plus an unsigned device build; identify remaining
  real-device acceptance work.
- Server behavior: Python tests and, when networking changes, a local authenticated
  request plus unauthorized-request check.
- Cross-cutting protocol changes: all Swift and Python tests.
- Documentation-only changes: link/path/command review; no build is required.
- Packaging changes: `bash -n scripts/package-altstore.sh` and a full packaging run.

## AltStore packaging rules

- When asked to package for AltStore, run `./scripts/package-altstore.sh`.
- The sole canonical artifact is `build/DawnPilot.ipa`. Do not add versions,
  dates, `final`, or numbered copies to the filename.
- Replace the canonical IPA only after all checks succeed. The script uses a
  temporary file and atomic replacement for this reason.
- The IPA must contain exactly one unsigned `Payload/DawnPilot.app` for arm64.
- It must not contain `_CodeSignature`, `embedded.mobileprovision`, `PlugIns`,
  `.appex`, `.env` files, credentials, device identifiers, or user runtime data.
- AltStore performs the final Apple ID signing. Unsigned status before handoff is
  expected and must be verified, not "fixed" with an unrelated local certificate.
- Report version, build number, minimum iOS version, file size, and SHA-256 when
  handing off an IPA.
- Never commit an IPA or upload it to GitHub Releases without explicit approval.
- `Config/ExportOptions.plist` is for a separate Xcode-signed export path; do not
  use it for the AltStore package unless the user explicitly requests that path
  and the signing team has been verified.

## Live deployment verification

After an authorized server deployment, verify all of the following without
printing the token:

- `dawnpilot.service` and Caddy are active, and the service listens only on
  `127.0.0.1:8787`.
- The public health endpoint returns HTTP 200.
- The existing unrelated website route still returns its expected response.
- An unauthenticated forecast request returns HTTP 401.
- An authenticated forecast request returns schema version `1`, the requested
  timezone, and non-empty hourly data.
- The service has no restart loop, and the persisted cache mode is `0600`.

## Real-device boundaries

Simulator and static IPA checks cannot prove AlarmKit behavior. State these as
remaining acceptance checks until verified on the target iPhone:

- AlarmKit authorization and actual ringing.
- Silent mode and Focus behavior.
- Nightly Shortcuts automation with "Run Immediately" and no confirmation.
- Apple Watch alert forwarding.
- AltStore installation, Apple ID app-ID limits, and seven-day refresh behavior
  for a free developer account.

## Git and change hygiene

- Preserve unrelated user changes and inspect the worktree before editing.
- Keep secrets, environment files, caches, build output, DerivedData, archives,
  provisioning profiles, and IPA files out of Git.
- Before committing, run `git diff --check`, inspect the staged file list, and
  scan staged content for credentials or private keys.
- Do not rewrite history, force-push, create a release, or change repository
  visibility without explicit authorization.
- In the handoff, report what changed, verification performed, artifact paths,
  and any real-device or external-system checks still outstanding.
